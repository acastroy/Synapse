Detailed Description

The core module.

This contains all the core functionality to control the synapse framework.

Here are some common sends that can be emitted by all event-handers of the core:

Sends: module_Error:
Sended to the requesting session if some error happend.
  • error A string describing the error.

synapse::SYNAPSE_REGISTER ( core_LoadModule  )

Dynamicly loads a synapse module.

  • path The absolute pathname of the .so file.
  • name Name of the module (will autodetermine the path)
Postcondition:
The module specified by path is loaded.
Sends: module_Init
To the initial session of the new module, so it can initalize itself.
Sends: module_SessionStart
To the initial session of the new module. (just like any other session that is started for a module)
  • username Username thats logged in to this session.
  • synapse_cookie Custom session cookie.
  • description Description of the session.
Broadcasts: module_SessionStarted
Tells the world a new session is started for this module.
  • session The session id of the new session.
Sends reply: modulename_Ready
Sended to all the sessions that requested the load, after the module indicates its ready. A module should always send a core_Ready when its ready.
  • session The original session id that sended the core_Ready.
Note:
If the module is already loaded, only a modulename_Ready is send.
synapse::SYNAPSE_REGISTER ( core_Ready  )

Indicates the src module is ready to be used.

Its mandatory for a module to call this when its ready.

Postcondition:
The module is marked as ready, so that a future core_LoadModule can return modulename_Ready immediatly.
Sends: modulename_Ready
to all modules that requested the module
  • session The default session of the ready module.
synapse::SYNAPSE_REGISTER ( core_Register  )

Registers an event handler.

  • event The name of the event.
  • hander (optional) The function name of the handler. If not specified, the name of the event will be used. A pointer to the function is looked up via the symbol table of the dynamicly loaded object.

You also can set a default handler that will be called for ALL events: Leave event empty and only specify handler.

Postcondition:
The handler will be called whenever a message with event is sended to the module.
Note:
Normally you would use the SYNAPSE_REGISTER macro, which calls core_Register automaticly.
synapse::SYNAPSE_REGISTER ( core_ChangeEvent  )

Changes the settings of an event.

  • event The event you want to change.
  • sendGroup The group that has permission to send the event.
  • recvGroup The group that has permission to receive the event.
  • modifyGroup The group that has permission to modify the permissions.
Postcondition:
The permissions are changed.
Note:
Messages that are already queued will be delivered without checking permissions again.
synapse::SYNAPSE_REGISTER ( core_Login  )

Changes the user of the src session.

  • username The username of the user you want to become.
  • password The password to authenticate you.
Postcondition:
The src session has changed user. (e.g. has different permissions)
Sends reply: module_Login:
To indicate a succesfull login.
  • username The username the session has become.
synapse::SYNAPSE_REGISTER ( core_NewSession  )

Starts a new session, with the same user as src.

  • username (optional)
  • password (optional) If specified will create a session with a different user instead of the src user.
  • maxThreads (optional) Max threads of new session.
  • synapse_cookie (optional) Cookie, can be used internally by module. This value is passed every time a handler for the session is called. It is passed as cookie parameter to the handler-function.
  • description (optional) description of the session
Postcondition:
A new session is created.
Sends: module_SessionStart:
to the new session, within the same module. This is to let the module know about its new session.
  • username Username of the session
  • description (optional) description of the session, usefull for debugging and administration.
  • (other parameters) Contains all specified arguments. (without password)
Broadcasts: module_SessionStarted:
to let the rest of the word know of the new session.
  • session The session id of the new session.
synapse::SYNAPSE_REGISTER ( core_Shutdown  )

Shuts down synapse.

Postcondition:
Deleted all sessions, including the core-session: You cant call any core-functions after sending this.
Broadcasts: module_Shutdown:
To let all modules know they should cleanup/shutdown and end all their threads.
Sends: module_SessionEnd:
To all sessions.
Broadcasts: module_SessionEnded:
For all sessions, to indicate to the rest of the world that the sessions have been ended.
  • session The session that has been ended.

synapse::SYNAPSE_REGISTER ( core_ChangeModule  )

Changes the settings of the src module.

  • maxThreads (optional) The maximum number of threads for the module. (default 1, e.g. single threaded)
  • broadcastMulti (optional) Set to 1 to deliver broadcasts to every session specific, instead of only to the default session of the module.
  • broadcastCookie (optional) Set to 1 to deliver broadcasts to every uniq session-cookie specific, instead of only to the default session of the module.
Postcondition:
Settings have been changed.

synapse::SYNAPSE_REGISTER ( core_Interrupt  )

Sends a thread.interrupt() to a executing call that was previously send from src.

It removes the call form the queue if its not executing yet.

  • event The event you want to interupt or cancel.
  • dst The destination session the event was send to.
Postcondition:
The event wont be delivered anymore, or a interrupt is send to the thread.
Sends reply: core_InterruptSent
To indicate the interrupt is send OR the call is removed.
  • pars The above specified parameters.
Note:
The thread must define boost interruption points for this to have effect.

synapse::SYNAPSE_REGISTER ( core_AddMapping  )

Adds a new event mapping.

  • mapFrom Event to map from
  • mapTo Event to map to
Postcondition:
Whenever event mapFrom is sent to destination -1, it will be remapped to mapTo. Multiple mappings per event are possible. Whenever the core maps an event, a core_MappedEvent is broadcasted. The end users should use mapper.html to remap events for things like remote controls or keyboard input.
Broadcasts: core_MappedEvent:
Informs of the new mapping status.
  • mappedFrom Event which got mapped
  • mappedTo List of target events we map to.
